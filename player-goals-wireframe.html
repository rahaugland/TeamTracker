Implementation Notes
Code Change Required
Currently GoalTracker.tsx gates the "Add Goal" button behind isCoach. To enable player self-service, remove that guard or add a second entry point that passes isCoach=false but still shows the create dialog. The PlayerGoalsPage already passes isCoach={false}, so the simplest fix is to always show the + button regardless of role.
Data Model
The existing player_goals table already supports everything shown here ‚Äî title, description, metric_type, target_value, current_value, deadline, is_completed, completed_at. For the "suggested goals" feature, no schema change is needed ‚Äî just pre-populate the form with suggested values derived from the player's current stats. The +1 quick update for custom goals calls updatePlayerGoal(id, { current_value: current + 1 }).
Auto-tracking
refreshGoalProgress() in player-goals.service.ts already handles auto-updating kill_pct, pass_rating, serve_pct, and attendance goals from live stats. The detail screen just needs to display this clearly with the "Auto-tracked" badge. The progress-over-time chart requires storing historical snapshots ‚Äî consider adding a goal_progress_log table or deriving from stat entry timestamps.
UX Patterns
Color-coded left borders on goal cards map to metric_type for instant visual scanning. Urgency indicators (red deadline badge when <7 days left) drive engagement. Progress ring summary gives an at-a-glance achievement overview. Bottom sheet for creation keeps context ‚Äî player sees their goals behind the form. Celebration screen on completion gamifies the experience and encourages setting the next goal.
RLS Policy
Ensure player_goals RLS allows players to INSERT/UPDATE/DELETE their own goals (where created_by = auth.uid() or player_id matches their player record). Currently the service uses the Supabase client which respects RLS ‚Äî verify the policy permits player-originated writes, not just coach writes.
